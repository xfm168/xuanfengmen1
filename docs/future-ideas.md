## 原则声明

> **Future Ideas 仅用于记录未来可能具有价值的方向，不代表必须实现，也不代表已经进入开发计划。**

Future Ideas 是：
- **未来可能性** — 有价值但当前未排期
- **探索方向** — 需要进一步研究验证
- **长期选项** — 可能永远停留在 Idea 状态

Future Ideas 不是：
- **待办事项清单** — 不代表必须完成
- **版本路线图** — 不代表已纳入计划
- **承诺交付** — 不代表对外承诺

任何 Idea 从 Idea → Accepted → Planned → Implemented，都需要经过：
1. Governance Freeze 四项收益检验
2. 资源评估与排期确认
3. 明确的 ROI 论证

**Rejected 状态同样重要** — 明确淘汰无效想法，避免反复讨论浪费精力。

## 生命周期与状态

### 状态定义

| 状态 | 含义 |
|------|------|
| **Idea** | 想法提出，待评估 |
| **Accepted** | 评估通过，确认有价值，待排期 |
| **Planned** | 已纳入路线图，计划实施 |
| **Implemented** | 已实现，进入正式版本 |
| **Rejected** | 评估不通过，明确淘汰（附原因） |
| **Archived** | 已归档，保留记录但不再跟进 |

### 状态流转

```
Idea → Accepted → Planned → Implemented
  ↓         ↓
Rejected  Archived
```

**Rejected 状态的重要性**：很多想法以后证明没有价值，应该明确淘汰，
避免反复讨论、浪费精力。Rejected 必须写明拒绝原因。

### IDEA-009：P0-② 子时换日策略模式

- **标题**：子时换日可配置策略（策略模式架构）
- **背景**：P0-① 评审建议 P0-② 不要仅实现单一流派，而是在架构层支持多种规则策略。
  子时换日是命理领域争议最大的基础规则之一。
- **目标**：通过策略模式统一实现多种换日规则，而非硬编码：
  - 晚子时换日（现代常见，23:00 后算次日）
  - 早子时换日（部分传统派，00:00 后算次日但子时跨两日）
  - 子初换日（23:00 子初即换日）
  - 午夜 00:00 换日（公历标准）
  - 可配置策略（高级用户自定义）
- **收益**：
  - 兼容不同流派，提升产品专业性
  - 架构可扩展，新增流派无需重构
  - 支持 Explain 标注流派差异
- **风险**：
  - 多策略维护成本上升
  - 用户困惑：默认策略是什么？
  - 需明确各策略的典籍依据与证据等级
- **优先级**：高（P0-② 核心设计）
- **预计版本**：P0-② 设计阶段即纳入
- **负责人**：待定

### IDEA-011：多层质量验证体系

- **标题**：开放式多层质量验证体系（替代固定四层）
- **背景**：P0-① 架构评审建议将"四层共同校验"抽象为"多层质量验证体系"，
  避免宪章因新增验证方式而频繁修改，延长宪章生命周期。
- **目标**：建立开放式验证体系，当前默认四层，未来可扩展：
  - ① 单元测试（Unit Test）— 函数级正确性
  - ② 回归测试（Regression）— 数据集不退化
  - ③ 权威历法对照（Benchmark）— 与专业软件对比
  - ④ 真实命例验证（Gold Case）— 真实案例验证
  - ⑤ AI Cross Validation（AI 交叉验证）— 未来扩展
  - ⑥ Master Review（专家审核）— 未来扩展
  - ⑦ User Feedback Validation（用户反馈验证）— 未来扩展
  - ⑧ A/B Accuracy Test（版本准确率对比）— 未来扩展
- **收益**：
  - 宪章生命周期延长，无需频繁修改
  - 验证体系可随项目发展持续演进
  - 新验证方式可平滑接入
- **风险**：
  - 需要设计验证层的注册与扩展机制
  - 过多验证层可能增加验收成本
- **优先级**：高（已写入宪章 6.1 节）
- **预计版本**：V4.8.1-pN（Patch 实现扩展机制）
- **负责人**：待定
- **状态**：Accepted（已写入宪章，待实现扩展机制）
- **来源**：P0-① 架构评审补充建议第一条

### IDEA-016：Governance Review（治理复核）

- **标题**：P0 Final 治理复核 — 保持最小、稳定、有效
- **背景**：P0 阶段最终架构评审建议，治理体系不应无限增长。
  P0 Final 阶段应增加一项治理复核，确认整个治理体系保持最小、稳定、有效。
  真正优秀的软件工程不是流程越来越复杂，而是在保证质量的前提下保持足够简单。
- **目标**：在 P0 Final 阶段执行治理复核，检查：
  - 是否存在重复文档（合并或删除）
  - 是否存在重复流程（合并或删除）
  - 是否存在重复规则（去重）
  - 是否存在长期未使用机制（删除或归档）
  - 是否存在可删除的治理成本（精简）
- **收益**：
  - 治理体系保持最小、稳定、有效
  - 避免流程负债累积
  - 让团队精力回归算法本身

  - 版本管理复杂度上升
  - 需设计多版本并行维护策略

### IDEA-021：算法成熟度等级（Algorithm Maturity）

- **标题**：统一算法成熟度标识体系
- **背景**：P0 治理最终评审建议，为每个算法建立成熟度标识，
  让用户一眼知道每个算法目前成熟到什么程度。
- **目标**：建立五级算法成熟度体系：

| 等级 | 含义 | 说明 |
|------|------|------|
| **Experimental** | 实验阶段 | 内部研发，不公开 |
| **Preview** | 预览阶段 | 公开测试，可能变更 |
| **Stable** | 稳定版本 | 接口稳定，可公开使用 |
| **Trusted** | 商业可信 | 经过验证，可商业使用 |
| **Verified** | 已验证 | 经过长期命例验证，最高可信 |

例如：
- 节气算法：Verified
- 子时换日：Stable
- 紫微斗数：Preview
- 格局：Stable
- 用神：Preview
- **收益**：
  - 用户一眼了解算法成熟度
  - 区分实验功能与稳定功能
  - 商业版本只发布 Trusted / Verified 算法
- **风险**：
  - 成熟度升级需要验证成本
  - 用户可能拒绝使用 Preview 功能
- **优先级**：高（核心算法透明化）
- **预计版本**：P1 阶段逐步建立
- **负责人**：待定
- **状态**：Idea
- **来源**：P0 治理最终评审建议第七条
