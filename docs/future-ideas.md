### IDEA-009：P0-② 子时换日策略模式

- **标题**：子时换日可配置策略（策略模式架构）
- **背景**：P0-① 评审建议 P0-② 不要仅实现单一流派，而是在架构层支持多种规则策略。
  子时换日是命理领域争议最大的基础规则之一。
- **目标**：通过策略模式统一实现多种换日规则，而非硬编码：
  - 晚子时换日（现代常见，23:00 后算次日）
  - 早子时换日（部分传统派，00:00 后算次日但子时跨两日）
  - 子初换日（23:00 子初即换日）
  - 午夜 00:00 换日（公历标准）
  - 可配置策略（高级用户自定义）
- **收益**：
  - 兼容不同流派，提升产品专业性
  - 架构可扩展，新增流派无需重构
  - 支持 Explain 标注流派差异
- **风险**：
  - 多策略维护成本上升
  - 用户困惑：默认策略是什么？
  - 需明确各策略的典籍依据与证据等级
- **优先级**：高（P0-② 核心设计）
- **预计版本**：P0-② 设计阶段即纳入
- **负责人**：待定
- **状态**：Accepted（已纳入 P0-② 设计参考）
- **来源**：P0-① 评审意见第六条

### IDEA-010：传统命理算法四层校验规范

- **标题**：传统命理算法四层共同校验发布标准
- **背景**：P0-① 评审建议将以下原则写入长期开发规范：
  传统命理算法必须同时经过 ① 单元测试 ② 回归测试 ③ 权威历法对照
  ④ 真实命例验证，四层共同校验后方可发布。
- **目标**：将四层校验纳入 Acceptance Gate 标准检查项，作为算法类模块的硬性门槛。
- **收益**：

  - 避免"编译通过 = 算法正确"的误区
  - P0-① 大寒 BUG 的经验制度化
- **风险**：
  - 增加每个模块的验收工作量
  - 真实命例库未建成前，第 ④ 层可能覆盖不足
- **优先级**：高
- **预计版本**：可从 P0-② 起执行（宪章 Acceptance First 原则的具体化）
- **负责人**：待定
- **状态**：Idea
- **来源**：P0-① 评审意见第二条

### IDEA-011：多层质量验证体系

- **标题**：开放式多层质量验证体系（替代固定四层）
- **背景**：P0-① 架构评审建议将"四层共同校验"抽象为"多层质量验证体系"，
  避免宪章因新增验证方式而频繁修改，延长宪章生命周期。
- **目标**：建立开放式验证体系，当前默认四层，未来可扩展：
  - ① 单元测试（Unit Test）— 函数级正确性
  - ② 回归测试（Regression）— 数据集不退化
  - ③ 权威历法对照（Benchmark）— 与专业软件对比
  - ④ 真实命例验证（Gold Case）— 真实案例验证
  - ⑤ AI Cross Validation（AI 交叉验证）— 未来扩展
  - ⑥ Master Review（专家审核）— 未来扩展
  - ⑦ User Feedback Validation（用户反馈验证）— 未来扩展
  - ⑧ A/B Accuracy Test（版本准确率对比）— 未来扩展
- **收益**：
  - 宪章生命周期延长，无需频繁修改
  - 验证体系可随项目发展持续演进
  - 新验证方式可平滑接入
- **风险**：
  - 需要设计验证层的注册与扩展机制
  - 过多验证层可能增加验收成本
- **优先级**：高（已写入宪章 6.1 节）
- **预计版本**：V4.8.1-pN（Patch 实现扩展机制）
- **负责人**：待定
- **状态**：Accepted（已写入宪章，待实现扩展机制）
- **来源**：P0-① 架构评审补充建议第一条

### IDEA-012：Gold Case 等级体系

- **标题**：真实命例库等级体系（Bronze / Silver / Gold / Diamond）
- **背景**：P0-① 架构评审建议将 Gold Case Library 规划为长期数据资产，
  建立等级体系而非简单的数量目标。
- **目标**：建立四级命例库等级体系：
  - **Bronze（青铜）**：1000 例 — 基础覆盖
  - **Silver（白银）**：5000 例 — 完整覆盖
  - **Gold（黄金）**：10000 例 — 深度覆盖
  - **Diamond（钻石）**：50000+ 例 — 专业级覆盖
  所有模块（八字、紫微、风水、奇门、择日）共享同一命例库。
- **收益**：
  - 命例库成为核心资产，而非临时数据
  - 跨模块共享，提升验证效率
  - 等级体系提供明确的建设目标
- **风险**：
  - 命例采集需大量人力和时间
  - 数据质量控制难度大（需验证出生时间准确性）
  - 存储与查询性能优化需求
- **优先级**：高（核心数据资产）
- **预计版本**：分阶段推进（与 P0/P1/P2 并行）
- **负责人**：待定
- **状态**：Idea
- **来源**：P0-① 架构评审补充建议第二条

### IDEA-013：RulePack 完整规则体系

- **标题**：完整规则体系架构（Rule Engine → RulePack → Snapshot）
- **背景**：P0-① 架构评审建议 RulePack 最终演进为完整规则体系，
  保证任何历史命盘 100% 可重现、可追溯、可验证。
- **目标**：建立三层规则体系：
  ```
  Rule Engine（规则执行框架）
        ↓
  RulePack（某一版本的规则集合）
        ↓
  Snapshot（生成命盘时的完整快照）
  ```
  - **Rule Engine**：规则执行、冲突检测、优先级计算的统一框架
  - **RulePack**：语义化版本管理的规则集合（major.minor.patch）
  - **Snapshot**：包含 RulePack 版本、算法参数、命盘结果的完整快照
- **收益**：
  - 历史命盘 100% 可重现、可追溯、可验证
  - 商业软件长期维护的核心能力
  - 算法变更透明可审计
- **风险**：
  - 需设计版本存储与迁移策略
  - 多版本共存的包体积与性能优化
  - 快照格式需稳定且可扩展
- **优先级**：中（重要但非紧急，P1 阶段可推进）
- **预计版本**：P1 阶段或 V4.8.1-pN
- **负责人**：待定
- **状态**：Idea
- **来源**：P0-① 架构评审补充建议第三条
