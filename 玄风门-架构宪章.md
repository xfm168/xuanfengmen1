# 玄风门 架构宪章 (Architecture Constitution)

> 版本：**V2.0** | 生效日期：2026-06-29 | 优先级：最高
>
> 本文档为玄风门项目最高开发规范。所有功能开发、重构、优化，都必须遵循。
> 不允许为了开发速度破坏整体架构。
>
> V2.0 更新：新增第十一原则「真实优于速度」、开发准则、开发纪律

---

## 总纲

玄风门不是一个普通命理网站，而是完整的AI玄学平台。

**核心竞争力：** 算法可信度 > 规则完整度 > 推导透明度 > 结果可验证

---

## 第一原则：算法第一

> 玄风门最大的竞争力不是UI，而是算法。

```
算法 > AI润色
```

### 具体要求

1. **AI只能解释算法，不能替代算法**
   - AI的作用：将算法结果转化为自然语言
   - AI不得：自行创造命理结论、绕过算法输出结果

2. **所有结论必须可追溯**
   - 每一个命理结论必须有对应的算法规则
   - 用户可以查看：命中了哪些规则、为什么得出这个结论
   - 支持：规则ID、规则名称、权重、推导过程

3. **算法质量优先于一切**
   - 宁可不发版，也要保证算法准确
   - 算法验证不通过，功能不得上线

---

## 第二原则：规则引擎统一

> 所有命理模块统一进入 Rule Engine。

```
所有模块 → Rule Engine → 统一输出
```

### 覆盖范围

| 模块 | 是否必须接入 | 备注 |
|------|-------------|------|
| 八字排盘 | ✅ | 基础数据 |
| 格局系统 | ✅ | 核心算法 |
| 喜用神 | ✅ | 核心算法 |
| 十神组合 | ✅ | 核心算法 |
| 神煞系统 | ✅ | |
| 大运流年 | ✅ | |
| 六亲分析 | ✅ | |
| 风水规则 | ✅ | 八宅/飞星等 |
| 缘分匹配 | ✅ | 合盘/匹配 |

### 统一输出

所有规则最终必须输出：

```typescript
interface RuleResult {
  confidence: number           // 可信度 0-100
  matchedRules: string[]       // 命中的规则ID列表
  matchedRuleDetails: {        // 规则详情
    ruleId: string
    ruleName: string
    priority: number
    weight: number
    matched: boolean
    reason: string
  }[]
  conflicts: {                 // 冲突规则
    rule1: string
    rule2: string
    resolution: string
  }[]
  reason: string               // 主要判断依据
}
```

### AI职责边界

- ✅ AI可以：读取规则结果，生成自然语言报告
- ❌ AI禁止：自行创造命理结论、绕过规则引擎输出

## 第三原则：模块完全解耦

> 任何模块不能互相直接依赖。

### 标准调用链

```
Calculator（调度器）
    ↓
Rule Engine（规则引擎）
    ↓
Analysis（分析聚合）
    ↓
AI（自然语言生成）
    ↓
UI（用户界面）
```

### 解耦要求

1. **每个模块可独立升级**
   - 升级格局系统，不影响喜用神
   - 升级风水算法，不影响八字
   - 替换AI模型，不影响算法层

2. **模块间通过接口通信**
   - 输入输出有明确的类型定义
   - 禁止直接访问内部状态
   - 禁止循环依赖

3. **Calculator只负责调度**
   - 不包含业务计算逻辑
   - 只做模块编排和结果聚合
   - 所有计算下沉到各模块

## 第四原则：Repository模式

> 页面不得直接访问Supabase。

### 标准分层

```
Page（页面）
  ↓
Service（业务服务）
  ↓
Repository（数据仓库）
  ↓
Supabase / API / LocalStorage
```

### 各层职责

| 层级 | 职责 | 禁止 |
|------|------|------|
| Page | 渲染UI、用户交互 | 直接调用Supabase |
| Service | 业务逻辑、结果组装 | 直接写SQL |
| Repository | 数据访问、缓存 | 业务逻辑 |
| Supabase | 数据存储 | —— |

### 好处

- 方便加缓存（Redis / 本地缓存）
- 方便数据库迁移（换数据库不用改业务）
- 方便API开放（对外提供API复用Service层）
- 方便企业版部署（私有化部署）

---

## 第五原则：Edge优先

> 核心算法逐步迁移到 Edge Functions。

### 目标架构

```
浏览器（前端）          Edge Functions          数据库
    │                       │                    │
    │  1. 用户输入          │                    │
    ├──────────────────────►│                    │
    │                       │  2. 算法计算       │
    │                       ├────────────────────►
    │                       │  3. 返回结果       │
    │  4. 展示 + AI解读     │◄────────────────────┤
    │◄──────────────────────┤                    │
```

### 好处

1. **保护算法源码** — 核心算法不暴露给浏览器
2. **统一计算口径** — 所有端用同一套算法
3. **支持付费API** — 可以对外提供API服务
4. **性能优化** — 利用Edge网络就近计算

### 迁移路线

| 阶段 | 迁移内容 | 版本 |
|------|---------|------|
| 第一阶段 | 八字排盘核心算法 | V7.5 |
| 第二阶段 | 格局/喜用神/十神 | V8.0 |
| 第三阶段 | 风水算法 | V8.0 |
| 第四阶段 | 全量算法迁移 | V8.5 |

## 第六原则：AI只是解释器

> AI负责解释，算法负责结论。

### AI的合法职责

| 职责 | 说明 |
|------|------|
| ✅ 解释算法结果 | 将规则输出转化为自然语言 |
| ✅ 润色表达 | 让报告更通顺、更易懂 |
| ✅ 生成完整报告 | 组织报告结构和语言 |
| ✅ 回答用户问题 | 基于已有结论回答疑问 |
| ✅ 个性化建议 | 基于算法结论给出建议 |

### AI的禁止行为

| 行为 | 说明 |
|------|------|
| ❌ 自行排盘 | 排盘必须由算法完成 |
| ❌ 自行判断格局 | 格局必须由规则引擎输出 |
| ❌ 自行判断喜用神 | 喜用神必须由规则引擎输出 |
| ❌ 自行判断旺衰 | 旺衰必须由规则引擎输出 |
| ❌ 编造命理术语 | 所有术语必须有依据 |

### 安全机制

```
用户输入 → 算法计算 → 规则输出 → AI读取 → 生成报告
                ↑                         ↑
                └────── AI不得跳过 ───────┘
```

## 第七原则：统一分析对象

> 所有分析模块统一返回 AnalysisResult。

### 统一接口

```typescript
interface AnalysisResult {
  score: number               // 综合评分 0-100
  confidence: number          // 可信度 0-100
  matchedRules: string[]      // 命中的规则
  matchedRuleDetails: RuleDetail[]
  conflicts: Conflict[]       // 冲突规则
  reason: string              // 主要判断依据
  summary: string             // 总结（算法生成，非AI）
  suggestion: string[]        // 建议列表
  derivationProcess: string   // 推导过程（供AI使用）
  moduleScores: {             // 各模块得分
    [moduleName: string]: number
  }
}
```

### 适用范围

所有产品线共用此结构：

- 玄风命理（八字、六爻、每日）
- 玄风风水（AI勘测、飞星、八宅）
- 玄风缘分（合盘、匹配）
- 玄风学院（学习评估）

### 好处

- AI只需要一套Prompt模板
- UI只需要一套展示组件
- 新增模块不需要重新设计架构
- 跨模块对比和聚合更容易

## 第十一原则：真实优于速度（Truth Over Speed）

> 玄风门的最高目标不是开发速度，而是算法可信度。

```
正确 > 完整 > 稳定 > 性能 > 速度
```

### 具体要求

1. **算法不能妥协**
   - AI 可以等待
   - 页面可以等待
   - 商业功能可以等待
   - 算法不能妥协

2. **禁止为了赶版本降低质量**
   - 算法验证不通过 → 延迟发版
   - 规则覆盖不完整 → 功能不上线
   - 推导过程不透明 → 继续完善

3. **宁慢勿错**
   - 宁可少做一个功能，不可错一个结论
   - 宁可晚一个版本，不可毁一世口碑
   - 真实可信是玄风门的生命线

### 算法四不原则

任何时候都必须遵守：

| # | 原则 | 说明 |
|---|------|------|
| 1 | **没有算法支撑的功能，不开发** | 先有算法，再有功能；不为了功能凑算法 |
| 2 | **没有测试的算法，不上线** | 单元测试+边界测试+经典案例验证，缺一不可 |
| 3 | **没有出处的Rule，不加入** | 每条Rule必须有典籍来源，不凭空创造规则 |
| 4 | **没有算法依据的AI解释，不允许输出** | AI只能解释算法结论，不能自行创造命理结论 |

> 玄风门未来最大的资产不是代码，而是算法体系。
> 代码可以重写，算法体系必须持续沉淀。

## 开发准则

每完成一个功能必须满足以下全部条件，缺一不可：

| # | 条件 | 说明 |
|---|------|------|
| 1 | ✅ 有规则 | 所有逻辑有明确的Rule定义，不是硬编码 |
| 2 | ✅ 有测试 | 单元测试 + 集成测试 + 边界测试 |
| 3 | ✅ 有文档 | 同步更新相关文档（规则/数据/接口/PRD） |
| 4 | ✅ 有历史记录 | 版本历史、修改记录、变更原因 |
| 5 | ✅ 可追溯 | 每个结论都能追溯到规则和数据来源 |
| 6 | ✅ 可回滚 | 出问题可快速回滚到上一版本 |
| 7 | ✅ 可解释 | AI能解释为什么得出这个结论 |

> 任何一个条件不满足，都不能视为真正完成。

V7.0 起，进入长期稳定开发阶段。

除重大问题外：

- ⛔ 不再调整架构
- ⛔ 不再新增长期文档
- ⛔ 不再大规模重构
- ✅ 优先完成既定路线图
- ✅ 所有开发围绕 Version Roadmap 执行

> 玄风门后续工作的重点只有一个：
> **持续提升算法质量和产品体验，直至形成行业领先的商业级 AI 命理平台。**

## 违反原则的处理

1. **代码审查不通过** — PR直接打回
2. **架构师一票否决** — 架构师有权否决任何违反原则的设计
3. **技术债登记** — 紧急情况下的临时方案必须登记技术债，限期重构
4. **定期审计** — 每季度进行架构审计，清理违反原则的代码
